## Part 2: UX / Layout Issues

### 2a. Confirmation Card Spacing

**Observed:** In the reflection conversation, when the AI proposes a system change (e.g., "Change anchor from X to Y"), the actionable card at the bottom has a large empty gap between the last AI message and the card.

**Fix:** The action card should render as a natural continuation of the conversation â€” tightly coupled to the message that proposed the change. Treat it as a special message type within the conversation flow, not a separate fixed-position element.

### 2b. Home Screen Habit Text Too Long

**Observed:** After the reflection updated the anchor, the home screen displays the full ritual as one quoted string:

> "Before bed OR first thing when I wake up, open bank app and log every transaction from today in a simple note or spreadsheet."

This is too long for an at-a-glance home screen. Takes more than 2 seconds to parse.

**Fix â€” Split anchor and action into separate visual elements:**

```
ANCHOR (smaller, secondary text color):
Before bed or when you wake up

ACTION (larger, primary text, the hero):
Log today's transactions in your note or spreadsheet
```

The full combined ritual text belongs on the System tab. The home screen should show a scannable version. Consider a max character limit for the home screen display (~80 chars for the action) and truncate with "See full ritual â†’" link to System tab if needed.

### 2c. "I did it earlier" vs "I can't today" Visual Weight

**Observed:** Both secondary actions below the primary CTA ("Mark today's rep") appear with equal visual weight.

**Fix:** These have different emotional valences:
- "I did it earlier" = positive (still a rep!) â€” should feel like a viable, easy action
- "I can't today" = miss path â€” should be available but visually recessed

Options:
- Make "I did it earlier" a subtle text button with the accent color
- Make "I can't today" a dimmer, tertiary-color text link
- Or: use a single row with "I did it earlier" left-aligned and "I can't today" right-aligned, with the former slightly more prominent

## Part 3: Reflection Conversation Quality â€” Root Cause Analysis

### What went wrong

The Week 1 Reflection conversation felt like talking to a neutral data analyst, not a coach celebrating a milestone. Specific issues:

1. **Not celebratory enough.** Week 1 completion is a big deal â€” the AI opened with statistics instead of acknowledgment. It should feel like a favorite coach pulling you aside after a strong first game.

2. **No domain awareness.** The AI knows the user is logging money transactions but demonstrated zero knowledge about the psychology of financial habit-building. No tips, no education, no connection to *why* this matters for their financial goals.

3. **No forward momentum.** The conversation didn't offer a micro-challenge or preview of what Week 2 looks like. The user had to drive toward "what's next" on their own.

4. **Opened with too much data.** "6 out of 7 daysâ€”8 in a row now. The difficulty is hovering around 4.3/5, which is higher than I'd want to see in Week 1." Three data points before the first question. A great coach leads with one observation and a question.

5. **Passive framing.** "Do you want to keep it exactly as is?" puts the burden on the user. The Reflection Partner should have a point of view.

### Root cause

The reflection prompt likely lacks three things:
1. **Full intake context** â€” the user's original goal, stated blockers, emotional tone
2. **Domain-specific coaching knowledge** â€” what's normal for Week 1 of a money habit? What gets hard in Week 2?
3. **Role-specific behavioral instructions** â€” the Reflection Partner needs a distinct emotional posture, not generic assistant behavior

## Part 5: Reflection Template System

### Concept

Each reflection milestone should have a defined emotional arc and required content blocks. The AI fills them conversationally, but the *shape* of the conversation is controlled.

This gives quality control over the conversation while preserving natural language delivery. It's the same pattern that makes the intake flow work â€” structured phases, conversational delivery.

### Week 1 Reflection Template

```yaml
milestone: week_1
emotional_tone: celebratory, warm, forward-looking
required_blocks:
  - type: celebration
    description: "Specific acknowledgment of Week 1 completion"
    must_reference: ["consistency", "original_goal", "domain_context"]
    
  - type: feeling_check
    description: "One subjective question about their experience"
    anti_pattern: "Don't ask about data â€” ask about feeling"
    
  - type: domain_insight
    description: "One relevant piece of domain knowledge"
    must_reference: ["habit_domain_knowledge"]
    
  - type: week_2_preview
    description: "Forward momentum with concrete micro-challenge"
    must_include: ["one_specific_action", "framed_as_building_on_momentum"]

optional_blocks:
  - type: system_adjustment
    trigger: "user raises friction OR data shows consistent miss pattern"
    delivery: "actionable card with Accept/Not now"
    
  - type: difficulty_exploration
    trigger: "user indicates habit is hard"
    approach: "curious, not concerned"
```

### Week 2 Reflection Template

```yaml
milestone: week_2
emotional_tone: grounded, observational, coaching
required_blocks:
  - type: pattern_recognition
    description: "Surface one pattern from 2 weeks of data"
    must_reference: ["specific_days_or_contexts", "trigger_patterns"]
    
  - type: threat_identification
    description: "What's the biggest threat to this habit continuing?"
    approach: "collaborative, not alarming"
    
  - type: toolkit_check
    description: "Is the environment supporting the habit?"
    may_suggest: ["setup_checklist_additions", "tiny_version_for_hard_days"]
    
  - type: week_3_preview
    description: "Week 3 focus: connecting the habit to a reward/insight"

  - type: system_adjustment
    trigger: "consistent friction point identified"
  - type: progression_preview
    trigger: "user asks about leveling up"
```

## Part 6: Domain Knowledge Injection

### The Problem

The AI currently has no domain-specific coaching knowledge. It knows the user logs money transactions but can't offer financial habit-building tips, anticipate Week 2 challenges, or suggest meaningful micro-challenges.

### Solution: Domain Knowledge Modules

Create a domain knowledge module for each supported habit domain. These get injected into AI prompts whenever the AI is acting in a coaching role (Reflection Partner, Recovery Coach, Progression Planner).

### Example Module: Financial Awareness

```markdown
## Domain: Financial Awareness / Money Management

### Common Week 1 Experiences
- Discomfort seeing real spending numbers
- Surprise at how much goes to small recurring purchases
- Feeling of control just from the act of observing
- Some days logging feels tedious, especially weekends with fewer transactions

### Common Week 2 Challenges
- Novelty wears off â€” "I already know what I spend on"
- Weekend logging drops (fewer structured transactions)
- Users start wanting to DO something about what they see (premature optimization)

### Week 2 Micro-Challenges (offer ONE)
- Categorize your transactions (food, transport, subscriptions, etc.)
- Identify your single largest daily expense each day
- Notice one "surprise" expense you didn't remember making
- Try logging BEFORE spending (just once) â€” does awareness change behavior?

### Common Week 3-4 Developments
- Patterns become visible (weekly spending rhythm, emotional spending triggers)
- Users naturally start making small spending adjustments
- The habit starts feeling less like a chore and more like a tool

### Key Psychological Principles
- Financial avoidance is emotional, not logical â€” awareness is the first intervention
- The "ostrich effect" (ignoring financial information) decreases with consistent, 
  low-stakes exposure
- Logging without judgment is critical early on â€” premature optimization kills 
  the awareness habit

### Obstacles & Responses
- "I already know what I spend on" â†’ "Knowing and tracking are different. 
  Tracking surfaces the patterns you can't see from memory."
- "This feels pointless" â†’ "The insight usually clicks around Week 3. 
  Right now you're building the data that makes insight possible."
- "I missed a day, should I go back and fill it in?" â†’ "If it's easy, yes. 
  If it feels like homework, skip it. Consistency beats completeness."
```

### Implementation Approach

For M1/M1.5, this can be a simple lookup â€” map the habit domain (identified during intake) to a domain knowledge markdown block, and inject it into the system prompt for coaching roles.

The domain should be stored as part of the habit definition:

```tsx
interface HabitDefinition {
  // ... existing fields ...
  domain: string;  // e.g., "financial_awareness", "sleep", "exercise", "reading"
}
```

For now, create domain knowledge modules for whatever habits are actively being tested. New domains can be added as needed. The key insight is that these modules exist once and get reused across all AI roles that need coaching context.
